Project finishing notes
=======================

Date: 2026-05-30

Context
-------

The root PDF describes an assignment to build a non-permissioned public ledger for auction transactions, with work split across a distributed ledger, secure P2P networking, and auction mechanisms. The existing repository already had a useful Rust foundation: Kademlia-style routing, gRPC/protobuf messages, RSA signing, bootstrap proof-of-work, auction structs, and simple blockchain structs. Several pieces were still unfinished or demo-only.

What I changed
--------------

1. Made protobuf generation self-contained.

   The original build required a system-level protoc installation. I added the protoc-bin-vendored build dependency and updated build.rs to set PROTOC automatically before tonic-build runs. This makes cargo test and cargo run easier for portfolio reviewers.

2. Fixed authenticated message signing.

   The previous sign_and_wrap implementation signed with Signer::new_without_digest, while verification used MessageDigest::sha256. That mismatch could cause signed RPCs to fail verification. I changed signing to RSA/SHA-256 so signing and verification use the same digest. I also added a test that generates an RSA key pair, wraps a PingRequest, and verifies it through extract_and_verify.

3. Implemented real local auction lookup for FIND_VALUE.

   The server-side find_value handler was stubbed: it returned a hard-coded value for key "found" and otherwise returned an empty closest-node list, then signed an unrelated string response. I replaced that with the expected DHT behavior:

   - Parse the requested auction key as an Address.
   - Check the node's local auction store.
   - If found, encode the protobuf Auction into FoundValue.value.
   - If not found, return the closest known nodes for the key.
   - Insert the requesting peer into the routing table after a valid request.
   - Sign a real FindValueResponse.

4. Fixed client STORE response verification.

   The client send_store_to_node path sent a StoreRequest but tried to decode the response as PingResponse. I changed it to decode StoreResponse.

5. Added client-side FIND_VALUE support.

   I added a find_value client helper that signs a FindValueRequest, calls the remote node, verifies the authenticated response, and returns a FindValueResponse. It is not wired to a user-facing CLI yet, but the protocol path now exists.

6. Added safe local auction retrieval.

   Node now has find_auction(&Address), which reads the synchronized auction set and returns the matching auction by key. This is used by FIND_VALUE and covered by tests.

7. Improved key management for multi-node demos.

   The old config selected ./keys/1, ./keys/2, etc. for a few hard-coded ports, but generated fallback keys as public_key.pem/private_key.pem in the repository root. That made multi-node demos easy to confuse and could overwrite identities. Config now derives key paths from the selected port:

   keys/<port>/private_key.pem
   keys/<port>/public_key.pem

   setup_keys creates the directory and generates missing keys there.

8. Added tests.

   Added focused tests for:

   - RSA/SHA-256 authenticated message round trip.
   - Stable auction key generation.
   - Auction protobuf round trip.
   - Node store/find auction behavior and duplicate suppression.
   - Block hash sensitivity to nonce changes.

9. Cleaned warnings and formatting.

   Ran cargo fmt across the project and removed unused helpers/imports that produced warnings. The remaining code builds cleanly under cargo test.

10. Added project documentation.

   Added README.md with project status, architecture, requirements, config, local run instructions, protocol summary, and honest portfolio notes about what is implemented versus prototype-level.

Important remaining work
------------------------

The project is now easier to build, test, explain, and demo, but it is still a prototype. The biggest remaining areas are:

1. Full blockchain consensus.

   The code has block hashing and proof-of-work utilities, but not a complete replicated chain, block gossip, fork-choice, mempool, or persistence layer.

2. Proof-of-reputation.

   The assignment mentions configurable PoW and proof-of-reputation. PoW challenge primitives exist; proof-of-reputation is not implemented.

3. Complete auction lifecycle.

   Auctions and bids can be represented and serialized, but bidding rules, bid validation, auction closing, winner selection, and settlement transactions still need to be added.

4. User-facing controls.

   The current application runs peer nodes. A CLI or HTTP API would make demos clearer: create auction, place bid, find auction, inspect routing table, and inspect local ledger.

5. Persistence.

   Routing state, auctions, keys, and ledger state are mostly in memory except for generated keys. A portfolio-grade next step would add a simple data directory and JSON or sled persistence.

Suggested next milestone
------------------------

The most valuable next milestone would be a small CLI around the existing peer:

    cargo run -- --port 8001 create-auction "Vintage keyboard" 50
    cargo run -- --port 8002 find-auction <auction_key>
    cargo run -- --port 8002 bid <auction_key> 75

That would turn the secure Kademlia prototype into a much more obvious portfolio demo without requiring a full consensus implementation first.
